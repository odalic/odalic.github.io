ADEQUATE : Story: User can propose new classes/entities for classification/disambiguation

Created by Tomas Knap, last modified on Nov 27, 2016

Github Issue: https://github.com/odalic/sti/issues/90 / https://github.com/odalic/odalic-ui/issues/24 / https://github.com/odalic/odalic-ui/issues/25

User Story

As a user i want to be able to propose new classes/entities for classification/disambiguation 

Preconditions

User can see results of classification/disambiguation (Story: User can see results of classification/disambiguation)

Detailed Description 

  1. User can click on Propose button in Step 2 (directly on the table) for the given cell
    1. As a result, dialog appears where he can define further aspects of the proposed resource
  2. When satisfied, user clicks OK and the system tries to create newly proposed concept in the primary KB
    1. If successfull, see Step 3. If not, warning is shown and user has to adjust the definition of the new concept
  3. User is returned back to the main table - he may see his new resource being used for the annotation. 
    1. All other previous annotations are removed. 
    2. Also the column is automaticall locked - see: Story: User may lock disambiguation/classification for the given cell

 

Alternative:

  1. User can open details of class/disambigiation in Step 2 (directly on the table)
    1. As a result, such dialog is open
  2. User can click on Propose button under the primary knowledge base section in the dialog
    1. As a result, dialog appears where he can define further aspects of the proposed resource
  3. When satisfied, user clicks OK and the system tries to create newly proposed concept in the primary KB
    1. If successfull, see Step 4. If not, warning is shown and user has to adjust the definition of the new concept
  4. User is returned back to the dialog - he may see his new resource being used for the annotation. 
    1. All other previous annotations are removed. 
    2. Also the column is automatically locked - see: Story: User may lock disambiguation/classification for the given cell

 

 


Acceptance Criteria & Test Scenarios

  1. User may propose new class
    1. Check that such class is really created in primary knowledge base
  2. User can immediately see the result of proposing the class - it is used for annotations
  3. User may propose new entity for disambiguation
    1. Check that such class is really entity in primary knowledge base
  4. User can immediately see the result of proposing the entity - it is used for annotations

Technical Notes: 

SPARQL Update query is used to interact with KB (standard sparql endpoint) - sparql graph protocol - https://www.w3.org/TR/sparql11-http-rdf-update/

Who does create URIs? Administrator of the primary KB via Odalic config defines base URL. User may then define suffixes. 

  • in the config file, one specifies 
    • the base URI for ontology (classes) 
    • the base URI for resources (entities)
    • the base conceptSchema used for proposed concepts
  • Possibility to define custom suffixes 

Creating new class:

  • new triples are constructed, which contain:
    • <classURI> 
      • a owl:Class
      • dcterms:title <label>
      • rdfs:label <label>
      • dcterms:title <altLabel>
      • rdfs:label <altLabel>
      • Optional: rdfs:subclassOf

Creating new entity:

  • new triples are constructed, which contain:
    • <entityURL>
      • a <classURI>
      • dcterms:title <label>
      • rdfs:label <label>
      • dcterms:title <altLabel>
      • rdfs:label <altLabel>

Searching for parent:

  • SPARQL is used to suggest the terms containing the search terms in their labels
    • By default we use the following predicates as holders for labels
      • http://www.w3.org/2000/01/rdf-schema#label
      • http://purl.org/dc/terms/title
    • But it can be configured int the KB config?
    • It shoudl search for owl:Classes
