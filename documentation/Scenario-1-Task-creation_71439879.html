ADEQUATE : Scenario 1 Task creation

Created by Tomas Knap, last modified by Václav Brodec on Mai 27, 2016

 

Initial State

The user is logged on the system (see: Scenario 0a Login and home screen)

Description

  1. User decides to create a new conversion task and the dialog in Mockup 6 appears, user fills the name and hits Next.

              a. As a result, system shows dialog in Mockup 1.

      2. User chooses the CSV file (Mockup 2).

              a. As a result, system stores the file(s).

      3. User optionally (he can click the Next right away) chooses the schema file (Mockup 5).

               a. As a result, system stores the file(s).

      4.User clicks “Next”.

               a. As a result, a dialogue depicted in Mockup 3 is shown.

      5. User chooses encoding, delimiters and escaping and optionally fills the file description (is pre-filled when schema file has been selected and contained the information).

               a. As a result, a dialogue depicted in Mockup 4 is shown.

      6. User decides to modify the prototypical configuration (Mockup 4). If he or she chooses to do so, the dialog from scenario 6a opens.

      7. User clicks "Run"

              a. As a result, system runs Odalic Core when available (otherwise puts it into queue and runs as soon as possible).

 

Alternatives

  • In 2. user may alternatively select location of a file (remote, HTTP) or local (file://). As a result, system caches the file when user clicks "Next" (3.) 
  • In 5. user can optionally override options from the configuration by setting the configuration and/or knowledge bases explicitly. 
    • setting configuration is described in Scenarios 6 a, b, c, d, e, f 
  • In 7. user can choose to run a preview first by clicking "Preview" button. Preview specific configuration options (limited amount or processed rows) are used when applicable.
  • At any point, user can "cancel" the creation of the task. In that case the task data is deleted. The files already uploaded are not deleted, however.

What can go wrong

  • System cannot obtain the specified input (file, configuration).
  • User chooses incorrect option for the file provided (eg. encoding; shows later and won’t be corrected by the application itself).
  • File cache service is unavailable.

System state on completion

  • Selected files to be used as inputs are cached for further use by Odalic Core algorithm.
  • Configuration is provided to the core and kept for refinement with feedback and optional storage at the end of conversion.

Requirements Motivating the Scenario

Implementation Notes

  • System must support caching of files

Another related issue is that if you wish to support processing tables with different schemas, or file formats other than CSVs, you need to allow users to configure a 'parser' that parses input binary documents and creates table object representations. As an example, the cast table on a IMDB webpage is different from that on a MusicBrainz webpage, hence different java classes need to be loaded accordingly to process them.
