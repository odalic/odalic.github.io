Firstly, the former TaskConfiguration has been split into two parts: a TaskConfigurationTemplate, which carries all the parts of configuration that are reusable between new tasks; and a usually much more short-lived TaskConfiguration, which serves as receipt for the Scheduler to actually create and plan the actual Task.

The TaskConfiguration thus contains (apart from the aforementioned template instance) fields describing the locations of output and input (as these are usually changed whenever you want to create a new task with the same settings as some existing task).





The Sequence diagram / Task configuration life-time starts with a user creating a new task (an event propagated to the task controller from GUI). If the configuration is based on another task, its TaskConfigurationTemplate is passed as initializing argument to the TaskConfiguration builder (shown as setter, but might be in the constructor as well), otherwise a blank one with no options filled is used. The builder is than used to keep track of all the settings that user selects using the GUI (only input definiton is presented to minimize the glut of various configuration options, but they all follow the same pattern).

 

When the user reaches the last step of configuration wizard and hits the Run button, the builder is used to build the TaskConfiguration, which is passed to the scheduler. The scheduler decides on the node used to run the provided configuration. After the node is assigned with a task born from the configuration, the algorithm (operating on the inputs and outputs through the -Content interfaces) starts processing.


Majority of the recently added interface methods found its way to the new class diagram.
