Latest changes are highlighted (blue color). 

CONTENT WRAPPING AND STAMPS

All regular API responses (which are of media type application/json) share the same format:

{ status: XXX, type: "YYY", payload: ZZZ, stamp: "client_defined_string" }

. The status contains an integer, the same as the code of the HTTP response itself. Type is either "DATA" OR "MESSAGE" string. When the type is "DATA", the payload contains a serialized domain object, which comprised the whole response in the previous versions of the API. When the type is "MESSAGE", than the payload has the following standard format:

{ text: "AAA", additionalResources: [ "BBB", "CCC", ... ], debugContent: "DDD" }

. The message itself is stored in the text, which may contain confirmation message or a message associated with an exception, depending on the status code of the response. The additionalResources is an array of URI strings, that may provide the client programmer with additional resources (usually documentation) to help him or her understand the message or debug its causes. The debugContent holds additional piece of information essential in debugging the call (usually stack trace) when the application server is in debug mode.

The text and debugContent can be null (or non-existent in the message), although in case of the text this is not recommended. The additionalResources cannot be null, but an empty array at most, when no additional resources can be provided.

The wrapping was introduced as a mean of making the interpretation of the response simpler for the client. The client can now determine the basic flow of processing of the response without the need to parser the payload.

The stamp field contains a string, that was sent to server by client in the request that originated the sent reponse. Clients can add an optional query parameter named "stamp" ("&stamp=client-set-string") with a value of their chossing to every request URL, and the server will send it back to them as a part of the response wrapping object.

To save space the following examples present data payloads in an unwrapped form, in which case the type is assumed to be "DATA" and the status code is as specified for the HTTP reponse.

When the client sends data to server, the wrapping is naturally not used. Also when the returned media type is not an application/json or when the returned content is a result of explicit export (such is the case of the annotated table), the response is also not wrapped and in case of error only the error status is set (as HTTP header) an no wrapped JSON message is sent.

Unless specified otherwise, the server can leave fields with null values from the result out.

HTTP ERROR CODES

The API follows standard conventions and returns 200 OK and 404 NOT FOUND where applicable. The 500 should be returned only when the server itself fails, if not, please report. Less common codes will be described in individual cases.

  • (1) PUT http://example.com/ files/{awesome_movies_2016}
  • (2) GET http://example.com/files/{awesome_movies_2016}

  • (3) DELETE http://example.com/files/awesome_movies_2016}
    • curl -X DELETE http://localhost:8080/odalic/files/awesome_movies_2016


  • (4) GET http://example.com/files/
  • (5)  PUT http://example.com/files/{awesome_movies_2016}/format
  • (6)  GET http://example.com/files/{awesome_movies_2016}/format
  • Query (1) uploads a file description (FS1) of a remote file or a cached local file, depending on the accept type (in which case (FS1)  does not apply, only the file itself is uploaded)
    • Identifier 'awesome_movies_2016' must be unique. Please note that the identifier can contain not only alphanumeric characters and underscore, but also spaces and other valid URL characters as long as they are property URL-encoded.
    • If the MIME type produced by the client is multipart/form-data, then the server expects only the file itself in the payload (see Screenshot 1) under input part label. 
    • If the MIME type produced by the client is application/json, then the server expects only the remote file description containing and assumes that the location is specified by the location attribute in the sent JSON object. The location can contain any valid URL.
    • The upload is non-resumable.
    • There is also available an alternative POST API call at ../files URL for the multiplart/form-data version, which derives the ID from the name of the uploaded file.
    • The format field is optional and allows for a remote file to immediately specify its format. Only the fields in the example (whose names are self-explanatory) are supported. The quoteCharacter, escapeCharacter, commentMarker can be set to null or omitted (in that case a default parser value is set or the parsing is automatic in that aspect). For a file that has been uploaded vie multipart/form-data, a default format (shown in the (FS1) example) is set and must be configured (if not suitable for the data) by a separate subsequent API call (see (5), (6)). The charset field value is expected to be a canonical name recognized by Java NIO (first column at https://docs.oracle.com/javase/8/docs/technotes/guides/intl/encoding.doc.html ).
  • Query (2) returns either a file description (FS2) for file with id 'awesome_movies_2016' or the file itself, depending on the MIME type set by the client.
    • If the MIME type is set to application/json, then the file description object is returned.
    • If the MIME type is set to text/csv, then the file specified in the location is provided. In case of remote location, the server attempts to download the file from the location and passes it as its own.
      • Alternatively, for clients that are unnable to provide appropriate Accept headere, they can get the data at http://example.com/files/{awesome_movies_2016}/data .
    • See (FS2) for output format of the file description. Notice the newly introduced cached attribute that distinguishes uploaded (cached) files from the remote ones.
  • Query (3) deletes the file (if local) and its corresponding file description.
    • Return HTTP 406 Conflict and an error message containing the IDs of the referring tasks, when the file is still utilized in some task configuration.
  • Query (4) returns a list of all the files. 
    • See (FS3) below for further details on the returned data.
  • Query (5), (6)
    • Only the fields in the payload example (FS4) (whose names are self-explanatory) are supported. The quoteCharacter, escapeCharacter, commentMarker can be set to null or omitted when put (in that case a default parser value is set or the parsing is automatic in that aspect). The charset field value is expected to be a canonical name recognized by Java NIO (first column at https://docs.oracle.com/javase/8/docs/technotes/guides/intl/encoding.doc.html ).

Tasks

  • (1) PUT http://example.com/tasks/{simple_task}
  • (2) GET http://example.com/tasks/{simple_task}
  • (3) DELETE http://example.com/tasks/{simple_task}
  • (4) PUT http://example.com/tasks/{simple_task}/configuration
  • (5) GET http://example.com/tasks/{simple_task}/configuration
  • (6) PUT http://example.com/tasks/{simple_task}/configuration/feedback
  • (7) GET http://example.com/tasks/{simple_task}/configuration/feedback
  • (8) GET http://example.com/tasks/{simple_task}/configuration/feedback/input
  • (9) PUT http://example.com/tasks/{simple_task}/execution
  • (10) DELETE http://example.com/tasks/{simple_task}/execution 
  • (11) GET http://example.com/tasks/{simple_task}/result
  • (12) GET http://example.com/tasks/result/rdf-export
  • (13) GET http://example.com/tasks/result/rdf-export/{turtle|json-ld|...}
  • (14) GET http://example.com/tasks/{simple_task}/state
  • (15) GET http://example.com/tasks/{simple_task}/result
  • (16) GET http://example.com/tasks/?states={false, true}&orderedBy={id,created} 
  • Query (1) creates a new task definition with the identifier 'simple_task'.
    • See (TS1) for the task description format. The id field is optional, but when filled, then it must be equal to the id provided in the URL.
    • The task is NOT KEPT in any persistent storage. After the task finishes or if the server is shut down, the Odalic core will simply forget about it.
  • Query (2) returns the task definition (TS1).
  • Query (3) removes the task. Although the task is deleted, any running computation might not be stopped immediately. 
  • Query (4) sets a new task configuration for possible next execution.
    • See (TS2) for the format description. The primary base has to always be set to one that is modifiable (see Bases).
    • The rowsLimit field when set to something else than null cuts of the input after the specified number of rows (the rowsLimit must be nonnegative when set and less than or equal to 2^31 - 1, which is also the maximum supported number of rows for the files). The uploaded files remain intact.
  • Query (5) returns the task configuration object (TS2).
    • The usedBases field must be a subset of the available bases. The primaryBase must be one of the used ones. The usedBases can be left out or set to null, in which case it defaults to all available bases.
  • Query (6) sets a feedback that the algorithm should take into account in case of next execution.
    • See (TS3) for the format description. Notice the changes:
      • The subjectColumnPositions are a map from KBs to column positions.
      • Chosen flag was removed and a separate chosen collection introduced along the existing candidates.
      • likelihood -> score
      • cellRelations removed
  • Query (7) returns the feedback set.
  • Query (8) returns the structured input of the task, upon which the feedback can be set.
    • See (TS4) for the format description.
  • Query (9) submits the task for execution.
    • Any previous executions must be either finished or cancelled.
    • The execution object format is described in (TS5), the draft flag is ignored for now. Returns 409 Conflict when the task is already running.
  • Query (10) cancels task execution. Returns 409 Conflict when the task has already finished.
  • Query (11) returns the result of task execution.
    • The result contains the annotations in structured format as described in (TS6).
    • The call waits for the process to finish.
  • Query (12) exports the result (when available) in a chosen (by the MIME type in the Accept header of the request) format.
  • Query (13) does the same thing as (12), but the format is a part of the URL (useful for browsers and other clients not supporting Accept headers).
  • Query (14) return the state of execution.
    • See (TS7) for format description. Notice the reduction and change of name of some states.
  • Query (16) returns a list of all tasks.
    • When the state query parameter is present and set to true, than the tasks aggregated with their states are provided: see (TS8) for format description.
    • The state parameter can be left out or set to false, then the results is just an array of individual tasks as described in (TS1).
    • When the orderedBy parameter is present and set to "id" or if it is missing, the tasks are sorted by their IDs in ascending order. This parameter can be also set to "created", then the tasks are sorted by creation time in descending order.

READY, RUNNING, SUCCESS, WARNING, ERROR

Possible states: READY, SCHEDULED, CANCELLED, FINISHED.

Need 5: READY (Created, not run yet), RUNNING, SUCCESS (result is ready), WARNING (not necessary; if result was returned but requires further user's attention), ERROR (no result generated; TM+ algorithm halted halfway)

CANCELLED - possibly not necessary

 RUNNING - transparent to user; a user shouldn't know whether the task is actually 'running' or just 'queued'

In more detail (mind you, this is just a proposal and you can change it as you deem appropriate):

  • READY: May be started, or removed. There is no result.
  • RUNNING: May be stopped. No other action is available. Stopping a task puts it into either 'READY', 'SUCCESS', 'WARNING' or 'ERROR' state. Depends on what was the last state before running the task.
  • SUCCESS: May be 'rerun' (removes result, puts into 'RUNNING' state), removed, cleared (removes result, puts into 'READY' state), or 'get result' may be applied.
  • WARNING: Same as 'SUCCESS' (therefore this state is not that much necessary, but may be useful in certain situations. What do you think?).
  • ERROR: May be 'rerun' (removes error message, puts into 'RUNNING' state), removed, cleared (removes error message, puts into 'READY' state), or 'get result' may be applied ('get result' presumably gives user only the error message).

REST API should be adapted accordingly. This is more of a UI's concern, but this decision does influence the API.

Open question: in which states can be a task configured? Only 'READY', or all of the 'READY', 'SUCCESS', 'WARNING', 'ERROR'?

'error message' (included in 'WARNING' and 'ERROR' states) should either be a part of 'result' (in this case format of the data should be slightly changed) or a new URL should be created for that purpose (e.g. GET http://example.com/tasks/simple_task/errors). I leave that up to you. István Satmári I have decided to make the warnings (arrays of warning messages a part of the Result (see above in the Result specification), but when the execution fails completely, than a single standard error message is sent with a 500 status code. Is that OK?

However, the error message should also contain either a human-readable text or an error number (so the UI can display its own variant of a human-readable text for the particular error). 

As for the API, possibly, these are influenced:

  • When a task is rerun, its previous result is automatically discarded after the new computation finishes (regardless whether successfully or not).
  • When a new computation is canceled, the task's result falls back to its last result (the previous result is discarded only if the computation actually finishes). (István Satmári This is not implemented yet)
  • The API should report the task is 'READY' when the result is removed/not present, otherwise it's 'SUCCESS'/'WARNING'/'ERROR'.
  • From the API's perspective 'feedback' is completely independent from the task's result and does not affect task's state in any way.
  • (Open question) Should reconfiguration of a task do something? (e.g. remove any previous result immediately).

Searching entities

  • (1) GET http://example.com/{base}/entities?query=Pra&limit=20
    • Returns array of entities (not discriminating classes, resources, properties,...) in the same format as in the Result (without the score naturally), see (ES1).
    • The base is the knowledge base where the search takes place.
       
    • The query is a URL parameter of the string searched for.
    • The limit is the maximum count of provided hits. Can be omitted and then 20 is used as default.
    • DEPRECATED.
  • (2) GET http://example.com/{base}/entities/classes?query=Cit&limit=20
    • Returns array of classes in the same format as in the Result (without the score naturally), see (ES1).
      • The "prefixed" field contains the same content as the resource field, if the prefix is not defined on the server for the resource URI.
        • Otherwise the prefixed contains the resource URI shortened by the use of prefix.
        • The prefix is always separated by a colon.
        • When sent to server, the prefixed field is optional and ignored by the server.
      • The "prefix" field contains an object consisting of "with" (the actual prefix string) and 'what" (associated URI) attributes. It is also optional and ignored when sent to server.
        • If not prefix defined, it is naturally null.
      • The "tail" field contains the rest of the string after the prefix separator, when the prefix is defined. Otherwise it is null. It is also optional and ignored when sent to server.
    • The base is the knowledge base where the search takes place.
    • The query is a URL parameter of the string searched for.
    • The limit is the maximum count of provided hits. Can be omitted and then 20 is used as default
  • (3) GET http://example.com/{base}/entities/resources?query=Pra&limit=20
    • Returns array of resources in the same format as in the Result (without the score naturally), see (ES1).
