ADEQUATE : User Feedback - possible actions of the user

Created by Tomas Knap, last modified on Nov 17, 2016

Mockups with user feedback actions described below: 

  • Feedback to subject column detection - Scenario 4a Customizing subject column - Selecting different subject column
  • Feedback to classificaiton/disambiguation - Scenario 4b Customizing classification class - selecting different non-suggested primary class for the column classification
  • Feedback to relation discovery - Scenario 4d Relations feedback

Locking of the suggested winning annotation (classification/disambiguation)

  • When the user is satisfied with the suggested winning annotation provided by Odalic Core, he may lock the suggested winning annotation

  • When the user adjusts certain annotation (classificaiton, disambiguation), then such adjustment is automatically locked. 

  • In UI, Step 2, there is an icon (of lock) for each table cell (further called also "annotation lock")

    • for the given column in case of lock next to the column header
    • for the given cell value in case of other cells

    • By default, all cells have the icon of the "open lock". User may click on that and as a result the icon changes to "closed lock"

      • When there is open lock (default), Odalic Core is in each execution trying to find best candidate for that cell/column/relations
      • When there is a closed lock, Odalic Core will take the value locked by the user and will not try to discover the value on its own. 

 

Apart from the lock at the level of table (see above), there are also locks at the level of knowledge base: 

  • when opening detail for the classification/disambiguation, he may lock the classfication/disambiguation at the level of knowledge base ("knowledge base annotation lock"). This behaves exactly the same as in case of annotation lock, the only difference is that such annotation is fixed only with respect to the locked knowledge base, so Odalic Core still tries to run normally for other knowledge bases.

Locking of the suggested winning annotation (relations)

  • In case of relations, use may lock the particular relation (in the particular knowledge base) - so there should be a lock icon when clicking on detail of the relation next to each particular relation. The motivation for that is that user would like to retain certain relation, but if the algorithm finds another relations between the same classes and within the same KB, that is OK (expected in most cases). 

Locking all disambiguation for the given classification 

Apart from the functionality above, there is also a different lock icon, which allows to lock the column classification together with all disambiguations.

Proposing new annotations (classification/disambiguation)

  • There is an icon, in Step 2, in each cell, which allows to propose new resource (not already available in the knowledge base) for the given cell classification/entity disambiguation. 
  • Such concept is always proposed in the primary knowledge base.  
  • There is also button propose in the detail of the classification/disambiguation. 

 

Proposing new annotations (relations)

  • In case of relations, use may draw new edge and then, in the detail, he may click propose. So there is no propose icon directly in the graph. 

 

Proposing new annotations (for all values in the given column)

Apart from the functionality above, there is also a different propose icon for the header cell, which allows to propose new entities disambiguating cell values in that column - it automatically creates entities for all cell values in that column which were not disambiguated/which were not disambiguated agains primary knowledge base.
