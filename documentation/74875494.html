User Story

As a user I am able to enter a new file for upload that shares identifier with one already existing. I expect the system to let me confirm my choice to overwrite the existing file with the new content and new meta-data.

After the upload is confirmed all tasks referring to the file use the updated version until the file is modified again or deleted.

I am able to overwrite a definition of existing remote file with a local one.

Preconditions

  • A file sharing the identifier already exists.

Detailed Description

  1. User commands the server to upload a new file, sharing identifier with already existing uploaded or remote file, using the UI or the API.
  2. Application displays a confirmation dialogue (API calls do not need a confirmation).
  3. User agrees to overwriting the file content and using the new version of meta-data.
  4. The old version of file is immediately replaced by the uploaded content. The upload is not confirmed until it is done so.
  5. User is informed about the successful update.
  6. When the user selects the file (by its identifier) later, the new meta-data are displayed.
  7. All subsequent usage of the file in referring tasks employs the new content and meta-data.

Acceptance Criteria & Test Scenarios

  1. Updated file content and meta-data remain available to the user in the same form even after server restarts and/or session renewal (if implemented).
  2. Restored file and/or its meta-data are identical to the one updated.
  3. Only a complete file content is updated.
  4. Only valid meta-data are updated.
  5. Both the file and existing meta-data are updated atomically, either both or none.
