ADEQUATE : Scenario 6a Task configuration - Knowledge bases

Created by Václav Brodec, last modified on Mai 21, 2016

Initial State

The logged-in user is either creating a task (from scratch or based on some pre-existing) or reviewing configuration of some scheduled task.

Description

  1. When the user selects Knowledge Bases tab, appropriate group appears (Mockup 1).

  2. User can select KBs from all available (either his/her own or public).

  3. User can optionally reorder used KBs. This affects the order of knowledge bases in classification dialogue. Also the first one is considered as default.

Alternatives

  • When the dialog is displayed during task creation, only Apply button is available.
  • When the dialog is displayed for already running task, only Apply in new task button is available.
  • If applied to already running task (which entered this state after dialog appearance), the dialog behaves as if Apply in new task button was clicked (notifying the user about it in the process).

What can go wrong

  • The KB is somehow (removed, restricted access) made unavailable (it is ignored, but user can be notified).

System state on completion

  • Used knowledge bases in currently edited task configuration are set.

Posted by brodecv at Jun 01, 2016 0:47

Agree. This is a very interesting problem where we can develop some innovative work. 

Generally there are two types of solutions, which I call 'in-process' or 'post-process'.

With 'in-process', we want to gather elements from all KBs, create feature representations for them, and apply algorithm to disambiguate (or classify or relation annotation), treating every element in the same way. However, because KBs can be unevenly populated, it is often the case that there are more features for KB1 but much less for KB2. As a result, the algorithm will always bias towards one KB (not sure in which way though until experiments are done). And im not sure how we can address this bias, it would be an interesting research problem.

With 'post-process', we simply run the algorithm on one KB at a time, then choose the best among all output. But the problem of unevenly populated KB will still surface. A score of 0.5 in KB1 could mean it is very high wrt KB1, but in KB2 it could be very low, or higher, or equally high, which we do not know.  

 

I guess in short term there is no easy way to address this, and you can possibly just adopt the post-process solution as it requires no change on the core tableminer+, and you can simply implement your post-process selection mechanism yourselves.

In long term, this is definitely an interesting and exciting research problem. Because if we can tackle multi-KB table annotation, effectively we are also providing a solution to KB alignment (sort-of).
