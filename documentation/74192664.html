ADEQUATE : Story: User remove all disambiguations in the given columm

Created by Václav Brodec, last modified by Tomas Knap on Sep 22, 2016

Github Issues: https://github.com/odalic/odalic-ui/issues/72 , https://github.com/odalic/sti/issues/29

Req: 75

User Story

As a user I sometimes want to keep a whole column ambiguous. It usually happens in the case of numeric columns or other where the disambiguation provides no benefit. I should have the possibility to remove all disambiguations. Then I can lock the disambiguations to prevent Odalic Core from disambiguating.

The choice is remembered for the re-runs, so unless I revert the choice back, all subsequent exports will keep the cell ambiguous.

The ambiguous cells may still appear in the result as an object in a triple consisting of a value from the subject column on the same row and a predicate determined by the class assigned to the column the ambiguous cell resides in.

When I decide leave a whole column ambiguous, individual disambiguation choices are ignored, but remembered in case I change my mind.

The option to leave a column ambiguous is controlled by a dedicated form element (checkbox). When left ambiguous, all individual cells controls are disabled, but their settings are kept until the next run in case I change my mind.

Making whole column ambiguous must not prevent either the algorithm from attempting to classify the column using only its header or the user doing it manually.

See Scenario 4b Customizing classification class - selecting different non-suggested primary class for the column classification - there is also button on the main table to prevent and remove disambiguations for the given column



Preconditions

  • The algorithm has been run at least once on the data or the application already knows how many columns are available in the input.

Detailed Description

  1. User indicates that all cells in column are to be left ambiguous (by un-checking associated check-box ).
  2. When the user chooses to export the result, the cells appear in the result only in its ambiguous form (without assigned entities and only as a literal objects).
  3. The choice is saved for the re-runs.
  4. The associated check-box is filled according to the choice when loaded in the next run.

Acceptance Criteria & Test Scenarios

  1. The choice to leave the cells ambiguous is also applied in the subsequent export.
  2. Leaving a whole column ambiguous supersedes any individual disambiguation choices.
  3. Associated form elements are pre-filled according to the set choice when loaded.
  4. Neither automatic nor manual classification is affected apart from taking the information from disambiguated cells away.

Variations

  • Optional goal: The user is able to take his decision back without any problems affecting the immediate export. No re-run is necessary (unless other feedback, requiring a re-run is provided).
