ADEQUATE : REST API: 1st iteration (first draft)

Created by István Satmári, last modified by Tomas Knap on Aug 03, 2016

Deprecated by REST API: 1st iteration

A proposal for REST API for 1st iteration.

This is a work in progress. Anybody can (and should ) edit the document; any useful info is very welcome.

- Istvan

Additional resources:

  • (R1) A useful resource on REST practices and recommendations: http://www.restapitutorial.com/resources.html
  • (R2) HTTP responses: http://www.restapitutorial.com/lessons/httpmethods.html 

Signing up, Authentication

  • Not necessary in the 1st iteration. However, if you see "GET http://example.com/users/John/files/awesome_movies_2016" and similar queries, note that the part "users/John" may be provided by a user, but should be ignored for now.
  • When designing classes, interfaces, etc., take into account that such queries will need to be checked against an ACL (access control list) later.
    • This will be probably done using an aspect programming.

Files

  • A service for uploading and downloading files. A file-hosting service, if you will.
  • (1) PUT http://example.com/users/John/files/awesome_movies_2016
  • (2) GET http://example.com/users/John/files/awesome_movies_2016
  • (3) POST http://example.com/users/John/files/ 
  • (4) DELETE http://example.com/users/John/files/awesome_movies_2016
  • (5) GET http://example.com/users/John/files/
  • Query (1) uploads a file. 
    • Identifier 'awesome_movies_2016' must be unique.
    • Returns HTTP 200 OK (see R2), if no problems arise.
    • An upload is non-resumable.
    • Uploading remote files directly, e.g. "http://site.com/my_awesome_table.CSV" is not possible.
  • Query (2) returns  a file with the corresponding identifier 'awesome_movies_2016'. 
    • The file is immediately downloaded upon the request.
    • If the file does not exist, a relevant HTTP reponse should be returned.
  • Query (3) ignored for now.
  • Query (4) removes the file with the corresponding identifier, should it exist.
    • A reponse, either "OK" or "Failure" must be returned.
  • Query (5) returns a list of all the files. 
    • See (FS1) below for further details on the returned data.
  • (8) GET http://example.com/files/
  • Query (8) ignored for now.
  • There has still no been specified relation in scenarios between input CSV files and their meta-information.
    • I suggest we drop the uploading of meta-information files and assume default CSV files only (at least for the first iteration)
    • i.e. only commas for separation of values, LF, and UTF-8

(FS1)

[ { Identifier: "awesome_movies_2016", UploadedOn: "YYYYMMDDHHMMSS", Format: "CSV", Owner: "Default" } , /* ... */ ]

JSON format, obviously. For the first iteration only the "Identifier" and "Format" is crucial (which should be easily provided by an underlying file system). Ignore the rest, or use placeholders.

Tasks

  • Everything related to manipulating and querying the tasks.
  • (1) PUT http://example.com/users/John/tasks/simple_task
  • (2) GET http://example.com/users/John/tasks/simple_task
  • (3) POST http://example.com/users/John/tasks/
  • (4) DELETE http://example.com/users/John/tasks/simple_task
  • (5) GET http://example.com/users/John/tasks
  • Query (1) creates a new task with an identifier 'simple_task'.
    • See (TS1) below for further details on the 'put-ed' data from a user.
    • A new task with the configuration specified in (TS1) is created.
    • The identifier 'simple_task' must be unique.
    • Do not forget about the HTTP responses.
    • The task is only created, but not immediately run.
  • Query (2) unresolved for now.
  • Query (3) ignored.
  • Query (4) removes the task.
    • If the task is running, it shall be stopped.
    • The task is removed from a database.
    • Associated resulting output file (RDF) is not removed.
    • Associated draft file is removed. (If applicable.)
  • Query (5) returns a list of all tasks created (and not yet deleted).
    • See (TS3) below for further details on the returned data.
  • (6) PUT http://example.com/users/John/tasks/simple_task/execution
  • (7) GET http://example.com/users/John/tasks/simple_task/execution
  • (8) DELETE http://example.com/users/John/tasks/simple_task/execution
  • Query (6) starts an execution of a task, if applicable.
    • Each task is associated with exactly 0..1 running executions.
    • Adequate response should be generated depending on if starting an execution is possible.
    • If the execution fails (e.g. an input file is missing), the information should be reflected in the task's state.
    • See (TS4) below for further details on the 'put-ed' data from a user.
  • Query (7) returns information about a state of the task.
    • See (TS5) below for further details on the returned data.
  • Query (8) stops an execution of the task, if applicable.
    • Again, do not forget about an appropriate response.
  • (9) PUT http://example.com/users/John/tasks/simple_task/configuration
  • (10) GET http://example.com/users/John/tasks/simple_task/configuration
  • Query (9) modifies the configuration of the task.
    • The data is identical to (TS1).
    • A running task may be modified as well, but the current execution will produce results based on the previous configuration.
    • Unresolved: Should only a part of the configuration change (e.g. feedback only), the query should be usable in the same way, but instead of providing all of the data specified in (TS1), only the required fields should appear. This is probably not crucial for iteration 1.
  • Query (10) returns the configuration of the task.
    • The data is identical to (TS1).
  • (11) GET http://example.com/users/John/tasks/simple_task/draft
  • (12) DELETE http://example.com/users/John/tasks/simple_task/draft
  • Query (11) returns a task's 'draft' (previously called suggestions).
    • If applicabled, i.e. the task has finished and produced the requested output.
    • See (TS6) below for further details on the returned data.
    • Note that the draft shall not be stored in the "Files Service". The draft is tied to the task.

    • There may be a need for a mechanism for querying only certain parts of the file. Something in a following way:

      • GET http://example.com/users/John/tasks/simple_task/draft?column="['col1', 'col2']"&cells="['10,30', '15,20']"

      • (GET parameters are more suitable for this type of query. Using 'slashes' would probably be quite restrictive.)

      • The data should probably still be in a format of (TS6) (just with less information).

  • Query (12) may remove the 'draft'.
  • (13) GET http://example.com/users/John/tasks/simple_task/result
  • (14) DELETE http://example.com/users/John/tasks/simple_task/result
  • Query (13) returns the resulting RDF.
    • If applicabled, i.e. the task has finished and produced the requested output.
    • The result shall be stored in the "Files Service".
    • The query (13) actually redirects to the "Files Service", e.g. GET http://example.com/users/John/files/simple_task_result
  • Query (14) may remove the result.
    • The query (14) actually redirects to the "Files Service",  e.g. DELETE http://example.com/users/John/files/simple_task_result 
  • (15) GET http://example.com/tasks/
  • Query (15) ignored.

(TS1)

{ InputFile: "http://general-url/even-to-our-rest-api-files-service", Feedback: { /* see TS2 */ } }

Ignored in this iteration: meta-information file, knowledge base, RDF export configuration, constraints on where to place an output.

(TS2)

{

  SubjectColumn: 7,

  Ignore: {           

      Columns: [ { Index: 1, Type: "NO-CLASSIFICATION" }, { Index: 2, Type: "NO-CELLS-DISAMBIGUATION" }, /* ... */ ],

      Cells: [ { Column: 9, Row: "127" }, /* ... */ ]

  } 

  Values: {

      Columns: [ { Index: 9, Type: "CHOOSE-OTHER", Value: "http://yumyumpedia.org/t/dish"}, { Index: 10, Type: "FORCE-VALUE", Value: "http://yumyumpedia.org/t/city" }, /* ... */ ],

      Cells: [ { Column: 9, Row: "127", Type: "CHOOSE-OTHER", Value: "http://yumyumpedia.org/t/pizza" }, /* ... */ ],

      Relations: [ { Column1: 9, Column2: 7, Type: "FORCE-VALUE", Value: "http://yumyumpedia.org/t/hasPopulation" }, /* ... */ ]

    }

}

Each of the exemplar fields may or may not be provided (e.g. we can ommit SubjectColumn, etc). 

We can assume the data will not be big, since it needs to be specified manually by a user.

In this iteration we assume existence only of a single knowledge base. (This setting is immutable in this iteration.)

BTW, this is just a proposal, if nobody comes up with a better idea.

(TS3)

[ { Identifier: "simple_task", CreatedOn: "YYYYMMDDHHMMSS",  State: "Running",  Result: null}, { Identifier: "great_task", CreatedOn: "YYYYMMDDHHMMSS",  State: "Finished",  Result: "great_task_result" }, { Identifier: "bad_task", CreatedOn: "YYYYMMDDHHMMSS",  State: "Failed",  Result: null}, /* ... */ ]

The returned data is 'compact', only the crucial info is returned.

Result contains identifier of a file in the "Files Service" produced by a corresponding task.

Note that the corresponding query (5) will have to support means for filtering and sorting the returned tasks! (And it is possible this will have to be done in the 1st iteration...)

(TS4)

[Draft, Export]

Uses a concept of 'flags.'

  • Draft: 
    • The task will execute and produce a so called 'draft'.
    • This requires man algorithm of TableMiner+ to run.
    • The 'draft' contains suggestions, disambiguated cells and many other important information for constructing a resulting RDF output.
    • If a draft was already produced, it is replaced after the execution finished.
  • Export: 
    • The task will execute and produce a resulting RDF output.
    • A draft must have been already produced.
    • If the task contains Export flag only and it has no Draft associated with, an error response shall be generated.
    • If a result was already produced, it is replaced after the execution finished.

Since the amount of information is relatively low, this could also fit into a GET parameter. What is the right way to do?

(TS5)

{ State: "Finished", Draft: true,  Result: "simple_task_result" }

Corresponds to (TS2). Draft as an indicator whether a draft is associated with the task (i.e. an execution finished and the draft has not been removed yet).

(TS6)

{

  SubjectColumn: 7,

  Values: {

      Columns: [ {Index: 9, Suggestions: ["http://adequate.at/concept/City", "http://adequate.at/concept/Country"]}, /* ... */ ],

      Cells: [ {Column: 9, Row: "127", Suggestions: ["http://adequate.at/concept/Moscow", "http://adequate.at/concept/Prague"]}, /* ... */ ],

      Relations: [ { Column1: 9, Column2: 7, Suggestions: ["http://adequate.at/concept/hasPopulation"]}, /* ... */ ]

    }

}

For the first iteration only 1 knowledge base is assumed, therefore this is not projected in the (exemplar) data.

Same applies as for the (TS2), this is just a proposal, if nobody comes up with a better idea.
