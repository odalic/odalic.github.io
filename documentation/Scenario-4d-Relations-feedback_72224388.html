ADEQUATE : Scenario 4d Relations feedback

Created by Josef Janoušek, last modified by Tomas Knap on Sep 23, 2016

Initial State

The user is logged on the system (see: Scenario 0 Login and home screen)

User have defined and run classification/disambiguation/relation discovery task (see: Scenario 1 Task creation). User is observing the results of the classification/disambiguation task.

Description

  1. The user may select any discovered relation and decide to customize

  2. User may specify alternative predicate for that relation

    1. Dropdown list with alternative suggestions for relation predicate computed by algorithm is opened.

      1. options are sorted by probability / score

  3. User may save such customization

    1. As a result, system depicts new relation in the table preview

Alternatives

  • In 2. when the user isn't satisfied with any suggested alternative, he selects "no suitable" option
  • In 2. when the user wants to provide new own predicate, he writes appropriate URL to the "custom" textbox
  • In 2, user may mark the given relation as being wrong

What can go wrong

System state on completion

The user provided alternative predicate of the relation.

Notes:

The original paper about TableMiner+ distinguishes two types of relations:

  • binary relations between the subject column and other NE-columns 

  • a property of the concept used to annotate the subject column to best describe data in the literal-columns 

    • property is considered both the annotation for the literal- column, and the relation between the subject column and the literal-column. 


In case of the mockup, both these relations (object or datatype properties) should be considered the same - they are relations/properties/predicates between columns A (a subject column) and B (target column).
