File Caching

Do we need to cache the file? If so, it is simply uploaded as if it was a local file on a user's computer. And yes, it would need to be explicitly deleted later. But the TableMiner+ may download the file itself right when needed, so what do you think?

 

We can skip the saving as long as the metadata will be also remotely provided and not altered. Otherwise we would have to introduce separate management of metadata files (and remember the source they belong to), which seems unnecessarily  complicated.
Treating all files the same way also brings some desired regularity.
To sum up I have not strong enough reasons to say let's do so and so, so decide as you wish or postpone the decision until the meeting.

File caching is not completely necessary, but useful: 1) As the user is providing feedback, Odalic Core may be reexecuted couple of times - it is not necessary to always download the file from the catalog IMHO. 2) The conversion may be executed in the future when the original source may have availability issues.

Options: (let's vote)

  • Core algorithm downloads the necessary input files right before executing a conversion. The file is disposed of right after the algorithm finishes. (Performance issues...?)
  • A user must upload each remote input file before executing a task on it. The file needs to be explicitly deleted later as well. (User friendliness...?)
  • Each input file is automatically temporarily downloaded to a cache. The file is deleted automatically after a given time (say, 7 days) as well. (Implementation: was the file for the given URI updated in the meantime? A separate component that checks the expiration date of file each 'n' hours?)

Req 20 - Should we allow user to just mark certain class/disamb/relation as being wrong?

In case of classification, I would say that user should provide alternative classification or can mark the column as not being classified. 

In case of disambiguation it is more difficult - user may see an obvious error there but maybe there is no corresponding entity in the target knowledge base and he does  not want to create new one - so he may just decide to drop certain suggestion

Regarding relation discovery, certain relation which is wrong may appear, so he may want just to delete the relation. 

Automatic propagation of feedback - how we will solve that in particular?

Automatically propagating user feedback at this scenario is fairly complex, and could require a fair bit of coding in tableminer+. In the first prototype maybe consider very basic strategies, or simply do not propagate. Examples:

  • if a winning class is changed, ideally it should trigger a check on the disambiguation results on all cells in that column, and for any cells whose tagged entity does not belong to the new class, they should be revised. Users may be given the option to allow this or not
  • following above, if any cells are revised, it could potentially affect the relations between columns. again users may be given the option to allow this or not
  • The same situation applies if users decide not to annotate the column

Classification - more concepts classifying the column? Disambiguation - more entities disamb.? How do we handle that in UI/export? Role of primary KB?

New requirements: Creation of new subject column? 

  • Summary: We do not allow to define new subject column as this would shift the tool more in the direction of general CSV editor, which is not what we are looking for. 

More knowledge bases - what happens if they do not detect the same subject column?

  • This may happen, e.g. the selection of the subject column also depends on the amount of entities being disambiguable.
  • Solution:
    • They have to agree on the same subject column?
      • This is simple, but may result in many false negatives when relevant yet differently focused KBs are used. (Václav Brodec)
    • Somehow combine the results allowing multiple subject columns.
      • Challenging, but I guess it might produce some interesting result. (Václav Brodec)

 

Angular v1 vs Angular v2

Is it worth to try Angular v2? 

  • I'd rather not. From my (albeit shallow) understanding, AngularJS 2 is a different beast and it would mean to scrap a lot of the UI code and basically learn everything from start. Then we could almost go with ReactJS (which ought to be simpler for views) + some solution for controllers. (Václav Brodec)
  • Definitely not. AFAIK, it is written in typescript which we do not use, has a different model of events similar to ReactJS, so a lot of time would be spent on re-learning things. (István Satmári)
